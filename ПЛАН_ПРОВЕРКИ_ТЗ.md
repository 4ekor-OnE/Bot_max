# Пошаговый план проверки проекта по ТЗ

Использование: выполняйте шаги **по порядку** там, где указана сквозная нумерация. В колонке **Результат** отмечайте ✓ / ✗ и кратко что увидели.  
Документ покрывает функции, реализованные в текущей сборке (`TZ_COMPLIANCE_REPORT.md`). Пункты **вне поставки** (webhook Flask, нагрузка 30 RPS) вынесены в конец.

**Нужно:** три аккаунта MAX (или последовательно с переназначением ролей), рабочий `.env`, запущенный `bot.py`.

---

## Часть 0. Среда и запуск

| Шаг | Что сделать | Ожидаемый результат |
|-----|-------------|---------------------|
| 0.1 | `pip install -r requirements.txt` | Установка без ошибок |
| 0.2 | Проверить `.env`: `BOT_TOKEN`, пароль админа | Файл на месте |
| 0.3 | Запустить `python bot.py` | Лог: БД инициализирована, polling, имя бота |
| 0.4 | *(Опционально)* на копии проекта: `python installer.py`, пройти мастер | `.env` создан, `get_me` успешен |
| 0.5 | Первый вход любым аккаунтом: `/start` | Меню по умолчанию (user), inline-кнопки |
| 0.6 | Тот же аккаунт: `/start@ИмяВашегоБота` (если поддерживается клиентом) | То же меню |

## Рекомендуемый порядок одного полного прогона

1. Часть **0** → **1** (админ готовит роли и справочники).  
2. Часть **2** (user) — создать обычную и срочную заявку.  
3. Часть **3** (support) — взять, комментировать, закрыть.  
4. Часть **4** (director) — все экраны и отчёты.  
5. Часть **5** (админ — заявки и очистка).  
6. Части **6–8** по необходимости.  
7. Часть **9** — зафиксировать в протоколе, если в ТЗ обязательны webhook/RPS.
