# MAX Support Bot

**Полная документация по проекту:** [DOCUMENTATION.md](DOCUMENTATION.md) · репозиторий: [github.com/4ekor-OnE/Bot_max](https://github.com/4ekor-OnE/Bot_max)

Чат-бот службы поддержки для платформы MAX: заявки, роли (пользователь / специалист / директор), админ-панель по паролю, уведомления, отчёты. Стек: **Python 3**, **SQLite**, **SQLAlchemy**, **maxapi**. В типовой поставке бот работает через **long polling** (`bot.py`); отдельный **Flask + webhook** из ТЗ при необходимости подключается как внешний слой.

## Быстрый старт

```bash
cd max_support_bot
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
cp .env.example .env
# Укажите в .env BOT_TOKEN (или MAX_BOT_TOKEN) и при необходимости ADMIN_PASSWORD / ADMIN_PASSWORD_HASH
.venv/bin/python bot.py
```

**Первичная настройка с мастером** (создаёт `.env`, вызывает `init_db`, проверяет токен через API):

```bash
python3 installer.py
```

### Веб-админка (Flask)

Отдельный процесс, **тот же пароль**, что для `/admin` в боте. Работает с **той же SQLite** (`DATABASE_URL`).

```bash
cd max_support_bot
.venv/bin/pip install -r requirements.txt
# В .env задайте WEB_ADMIN_SECRET_KEY (длинная случайная строка для cookie-сессий)
.venv/bin/python -m web_admin
```

По умолчанию: **http://127.0.0.1:5000** → перенаправление на `/login`. После входа: сводка, пользователи (роли), магазины, категории, заявки (фильтры и смена статуса).

Переменные: `WEB_ADMIN_HOST`, `WEB_ADMIN_PORT`, `WEB_ADMIN_SECRET_KEY`, опционально `WEB_ADMIN_DEBUG=1`. Для доступа из интернета — **только за reverse proxy (HTTPS)** и ограничение по IP/firewall; не открывайте порт напрямую без TLS.

На **VPS / своём ПК** достаточно venv и `bot.py` под systemd/supervisor. На **Jino** и аналогах обычно задают переменные окружения в панели и запускают тот же `bot.py`; путь к данным и SQLite см. `DEPLOY.md`.

## Команды

- `/start` — главное меню (поддерживается также `/start@ИмяБота`).
- `/admin <пароль>` — панель администратора без роли в БД: после успешного входа сессия `admin_mode` сохраняется в FSM; пользователь при необходимости **создаётся в БД автоматически**, чтобы inline-разделы админки работали.

Для роли **user** в меню есть **«🚨 Срочная заявка»**: после выбора магазина категория назначается автоматически («Касса и оплата»), приоритет — высокий, срок SLA — из настроек (`urgent_sla_hours` в «Настройки» админки). Обычное создание заявки по-прежнему через «Создать заявку» с выбором категории.

Пароль: переменная `ADMIN_PASSWORD` (разработка) или `ADMIN_PASSWORD_HASH` — SHA-256 в hex (нижний регистр), для production.

## Соответствие ТЗ (кратко)

- FSM создания заявки: после описания и фото — **экран подтверждения** («Подтвердить» / «Отмена»), затем запись в БД и уведомление специалистам.
- **Комментарии** к заявкам (`ticket_comments`), ввод специалистом из карточки заявки; системные записи при смене статуса; **уведомления** автору и участникам (см. `services/notification_service.py`).
- Специалисты: push о **новых заявках** (общий канал + режим «только срочные» в «Уведомления»), отдельные поля `notify_new_tickets` / `notify_urgent_only`.
- Директор: **отчёт за период** — выбор формата **CSV** / **Excel (.xlsx)** / оба, затем ввод двух дат `ГГГГ-ММ-ДД`.
- Лимит текста полей: **4000** символов (`MAX_MESSAGE_TEXT_LENGTH` в `config.py`).
- **`installer.py`** — интерактивная подготовка `.env` и проверка токена; **Flask + HTTPS webhook** в репозитории не поднимается — только long polling, пока не добавлен отдельный сервис.

## Админ-панель (`/admin <пароль>`)

После входа доступны (inline):

- **Пользователи** — список с пагинацией, поиск по ID / max_id / имени, смена роли (user / support / director).
- **Магазины** — список, добавление (текстовое имя), переименование, удаление (если нет заявок).
- **Категории** — список, создание (название → описание → SLA в часах), правка полей, удаление (если нет заявок).
- **Все заявки** — список с **фильтрами** (статус, магазин, исполнитель, период по дате создания), пагинация; карточка заявки, смена статуса, **назначение специалиста** (с уведомлением автора и исполнителя).
- **Документы** — загрузка txt/pdf/docx: название текстом, затем файл вложением; список, переименование, удаление.
- **Настройки** — SLA срочных/обычных по умолчанию, сроки хранения (заявки, сессии, логи*, temp-файлы) в таблице `system_settings`.
- **Очистка** — удаление старых решённых заявок, сессий FSM, файлов в `data/temp` по порогам из настроек.

\*Поле «логи» зарезервировано под будущий журнал; скрипт `cleanup.py` выполняет те же действия, что кнопки в админке.

Переменные окружения (опционально): `INSTRUCTIONS_DIR`, `TEMP_FILES_DIR`, `INSTRUCTION_MAX_BYTES`.

Ввод текста в админке обрабатывается только при активном шаге (поиск, название магазина и т.д.).

### Скрипт очистки (cron)

```bash
.venv/bin/python cleanup.py --all
# или по отдельности: --tickets --sessions --temp
# суточный агрегат в таблицу statistics (UTC):
.venv/bin/python cleanup.py --stats-yesterday
# .venv/bin/python cleanup.py --stats-day 2025-03-01
```
